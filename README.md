# PolygonSite-ML

ML-песочница для симулятора перехвата дронов. Этот репозиторий отделен от базового `PolygonSite`, чтобы можно было менять архитектуру, добавлять обучение и экспериментировать с управлением дронами без риска сломать исходный вариант.

## Текущая постановка

Обучаем только атакующие дроны. Антидроны, камеры, обнаружение и перехват остаются rule-based и используются как фиксированная среда.

Дроны обучаются не по одиночке, а как группа. Основной ML-контракт - централизованная командная policy: она видит состояние всей сцены и за один шаг выдает действия для всех дронов. Это позволяет учить распределение целей, отвлечение антидронов, совместные маршруты и другие групповые стратегии.

Стартовая расстановка тоже входит в обучение. До начала эпизода командная policy получает deployment-наблюдение и выбирает точки появления дронов на периметре полигона. Случайная расстановка остается только fallback-режимом.

Это важно для первого этапа: мы не обучаем две стороны одновременно, а сначала делаем управляемую среду, где командная ML-политика дронов пытается проходить существующую оборону.

## Архитектура

```text
.
+-- index.html                  # разметка интерфейса
+-- styles.css                  # стили
+-- src/
    +-- app.js                  # связывает UI, симуляцию и рендер
    +-- constants.js            # константы и параметры
    +-- math.js                 # геометрия, углы, rng
    +-- simulation.js           # состояние и шаг симуляции
    +-- render.js               # Canvas-отрисовка
    +-- policies/
        +-- dronePolicy.js      # rule-based и ML-ready policy для дронов
        +-- teamMlStub.js       # проверочная командная policy без реальной модели
    +-- ml/
        +-- observations.js     # кодирование состояния для ML
+-- scripts/
    +-- smoke-test.mjs          # headless-проверка симуляции
```

## Запуск

Для браузера нужен статический сервер, потому что проект использует ES-модули.

```bash
npm run serve
```

После запуска откройте:

```text
http://localhost:8000
```

Headless-проверка без браузера:

```bash
npm run smoke
```

## Управление на карте

- `Новый сценарий` перезапускает симуляцию с текущими параметрами.
- Левая панель содержит параметры симуляции, статистику и легенду.
- Правая панель содержит все, что относится к ML: выбор policy, scope обучения, размеры observation/action и seed эпизода.
- `ML Policy` переключает режим дронов: `Rule-based` или `Team ML Stub`.
- Колесо мыши меняет масштаб карты.
- ПКМ + перетаскивание перемещает карту.

## Что уже подготовлено для ML

- Симуляция отделена от Canvas-рендера.
- Есть `Simulation.reset()`, `Simulation.step()`, `Simulation.getState()`.
- Для дронов выделен policy-слой.
- Антидроны оставлены в `simulation.js` как фиксированная rule-based среда.
- Добавлен deployment-контракт для выбора стартовых точек на периметре.
- Добавлен deployment encoder: `encodeDeploymentObservation(sim)`.
- Добавлен командный encoder наблюдения: `encodeTeamObservation(sim)`.
- Добавлен ML-ready policy-класс `TeamWaypointMlDronePolicy`.
- Добавлен проверочный `Team ML Stub`, который использует тот же deployment/action контракт, что и будущая модель.
- Одиночный `encodeDroneObservation(sim, drone)` оставлен для debug и будущих decentralized-экспериментов.
- Добавлен детерминированный seeded RNG для воспроизводимых сценариев.

## Выбранная модель управления дроном

Эпизод теперь состоит из двух уровней решений.

Первое решение - расстановка:

```js
[
  d1_perimeter,
  d2_perimeter,
  // ...
  d15_perimeter
]
```

Каждое значение нормализовано в диапазоне `[-1, 1]` и декодируется в конкретную точку на периметре карты. Так модель может учить, откуда группе выгоднее начинать атаку.

Второе решение - движение. На этом уровне выбран не PID, а waypoint/steering-контроллер.

ML-политика должна выдавать массив действий для всех слотов дронов:

```js
[
  d1_dx, d1_dy, d1_intent,
  d2_dx, d2_dy, d2_intent,
  // ...
  d15_dx, d15_dy, d15_intent
]
```

Где `dx`, `dy` и `intent` нормализованы в диапазоне `[-1, 1]`. Декодер превращает `dx`, `dy` в локальные целевые точки для конкретных дронов, а `intent` задает тактическое решение: отрицательное значение - уклонение, ноль или положительное - атака. Штатная кинематика симуляции уже ограничивает поворот, скорость и движение.

Причина выбора:

- action space маленький и удобный для обучения;
- командная policy может координировать всех дронов сразу;
- модель не обязана учить низкоуровневую физику поворота;
- текущий рендер и антидроны не нужно менять;
- позже можно заменить только policy, не переписывая симуляцию.

PID-контроллер пока не нужен. Он станет полезен, если мы добавим более физичную модель движения: инерцию, перегрузки, задержки рулей, ограничение ускорения или управление заданным курсом. В текущей кинематике PID добавит лишние коэффициенты и усложнит обучение.

## Возможные следующие шаги

1. Сделать сбор командных эпизодов с deployment-действиями для imitation learning или offline RL.
2. Подключить ONNX Runtime Web или внешний Python-сервер.
3. Добавить командную reward-функцию для RL:
   - плюс за поражение цели;
   - минус за перехват;
   - минус за лишнее время;
   - бонус/штраф за качество стартовой расстановки;
   - бонус за одновременную атаку несколькими дронами;
   - бонус за поражение базы.
4. Добавить быстрый headless runner для тысяч эпизодов без Canvas.
